# RenaceCHESS Dataset Format

This document describes the dataset format, build process, and deterministic rules for RenaceCHESS.

## Overview

RenaceCHESS datasets are collections of chess position evaluations stored in **JSON Lines (JSONL)** format. Each line contains a complete **Context Bridge payload** representing a single position with its human-conditioned move probabilities, WDL estimates, and difficulty metrics.

## Dataset Structure

A dataset consists of:

1. **Shard files** (`shards/shard_XXX.jsonl`): JSONL files containing position records
2. **Manifest file** (`manifest.json`): Metadata describing the dataset, shard references, and split assignments

### Shard Format

Each shard is a **JSON Lines** file where:
- Each line is a complete JSON object (Context Bridge payload)
- Lines are newline-delimited
- No trailing spaces
- Canonical JSON formatting (sorted keys, no whitespace)

Example shard line:
```json
{"position":{"fen":"rnbqkbnr/pppppppp/8/8/8/8/PPPPPPPP/RNBQKBNR w KQkq - 0 1","sideToMove":"white","legalMoves":["a2a3","a2a4",...]},"conditioning":{"skillBucket":"1200-1400","timePressureBucket":"NORMAL","timeControlClass":"blitz"},"policy":{"topMoves":[{"uci":"e2e4","p":0.45},...],"entropy":3.2,"topGap":0.15},"humanWDL":{"pre":{"w":0.55,"d":0.15,"l":0.30},"postByMove":{...}},"hdi":{"value":0.65,"components":{...}},"narrativeSeeds":[],"meta":{"schemaVersion":"v1","generatedAt":"2024-01-01T12:00:00","inputHash":"..."}}
```

### Manifest Format

The manifest is a JSON file conforming to the Dataset Manifest schema. **M03+ uses v2 by default** (v1 is maintained for backward compatibility).

#### Manifest v2 (Current)

```json
{
  "schemaVersion": "v2",
  "createdAt": "2024-01-01T12:00:00",
  "shardRefs": [
    {
      "shardId": "shard_000",
      "hash": "sha256-hex-hash",
      "path": "shards/shard_000.jsonl",
      "records": 10000
    },
    {
      "shardId": "shard_001",
      "hash": "sha256-hex-hash",
      "path": "shards/shard_001.jsonl",
      "records": 5432
    }
  ],
  "assemblyConfigHash": "sha256-hex-hash",
  "datasetDigest": "sha256-hex-hash",
  "inputs": [
    {
      "type": "ingest_receipt",
      "digest": "sha256-hex-hash",
      "receiptId": "lichess-2024-01",
      "path": "/path/to/receipt.json"
    }
  ],
  "assemblyConfig": {
    "shardSize": 10000,
    "maxGames": null,
    "maxPositions": null,
    "startPly": 0,
    "endPly": null
  },
  "splitAssignments": {
    "train": ["shard_000", "shard_001"],
    "val": ["shard_000", "shard_001"],
    "frozenEval": ["shard_000", "shard_001"]
  },
  "filterConfigHash": "sha256-hex-hash"
}
```

**Key v2 fields:**
- `assemblyConfigHash`: Hash of assembly parameters (shard size, filters, etc.)
- `datasetDigest`: Stable dataset identity (hash of assembly config + input digests + schema versions)
- `inputs`: List of input sources (receipts or PGN files) with their digests
- `assemblyConfig`: Complete assembly configuration
- `shardRefs[].records`: Record count per shard

#### Manifest v1 (Legacy)

v1 manifests are still supported for backward compatibility but are no longer generated by default.

## Building Datasets

### CLI Command

#### Building from PGN Files (Backward Compatible)

```bash
renacechess dataset build \
  --pgn <file-or-dir> \
  --out <output-dir> \
  [--shard-size N] \
  [--max-games N] \
  [--max-positions N] \
  [--start-ply X] \
  [--end-ply Y]
```

#### Building from Ingest Receipts (Recommended)

```bash
renacechess dataset build \
  --receipt <receipt-file> [--receipt <receipt-file> ...] \
  --out <output-dir> \
  [--shard-size N] \
  [--cache-dir <cache-dir>] \
  [--max-games N] \
  [--max-positions N] \
  [--start-ply X] \
  [--end-ply Y]
```

**Arguments:**
- `--pgn`: Path to PGN file or directory (can be specified multiple times). Mutually exclusive with `--receipt`.
- `--receipt`: Path to ingest receipt JSON file (can be specified multiple times). **Recommended for auditable provenance.** Mutually exclusive with `--pgn`.
- `--out`: Output directory for shards and manifest
- `--shard-size`: Maximum number of records per shard (default: 10000). Minimum: 1.
- `--cache-dir`: Cache directory for resolving relative receipt paths (optional, defaults to receipt directory)
- `--max-games`: Maximum number of games to process (global limit across all inputs)
- `--max-positions`: Maximum number of positions to process (global limit)
- `--start-ply`: Start processing from this ply number (inclusive, default: 0)
- `--end-ply`: Stop processing at this ply number (exclusive)

**Examples:**

```bash
# Build from PGN files (backward compatible)
renacechess dataset build \
  --pgn games/2024/ \
  --out datasets/2024-jan/ \
  --shard-size 10000 \
  --max-positions 100000 \
  --start-ply 2 \
  --end-ply 50

# Build from ingest receipts (recommended for provenance)
renacechess dataset build \
  --receipt ingested/lichess-2024-01.json \
  --receipt ingested/lichess-2024-02.json \
  --out datasets/2024-q1/ \
  --shard-size 10000 \
  --cache-dir ~/.renacechess/cache
```

### Programmatic API

#### Building from PGN Files

```python
from renacechess.dataset.builder import build_dataset
from renacechess.dataset.config import DatasetBuildConfig
from pathlib import Path
from datetime import datetime

config = DatasetBuildConfig(
    pgn_paths=[Path("games/sample.pgn")],
    output_dir=Path("datasets/sample/"),
    shard_size=10000,  # Records per shard
    max_positions=1000,
    start_ply=0,
    end_ply=40,
)

# Use frozen timestamp for deterministic outputs
build_dataset(config, generated_at=datetime(2024, 1, 1, 12, 0, 0))
```

#### Building from Ingest Receipts

```python
from renacechess.dataset.builder import build_dataset
from renacechess.dataset.config import DatasetBuildConfig
from pathlib import Path
from datetime import datetime

config = DatasetBuildConfig(
    receipt_paths=[
        Path("ingested/lichess-2024-01.json"),
        Path("ingested/lichess-2024-02.json"),
    ],
    output_dir=Path("datasets/2024-q1/"),
    shard_size=10000,
    cache_dir=Path.home() / ".renacechess" / "cache",  # Optional
    max_positions=100000,
)

build_dataset(config, generated_at=datetime(2024, 1, 1, 12, 0, 0))
```

## Deterministic Rules

### File Processing Order

1. **PGN files are processed in sorted order** (lexicographic by path)
2. **Games within a file are processed in file order**
3. **Positions within a game are processed by increasing ply number**

### Position Filtering

- `start_ply`: Applied uniformly to all games (positions before this ply are skipped)
- `end_ply`: Applied uniformly to all games (positions at or after this ply are skipped)
- Limits (`max_games`, `max_positions`) apply globally across all inputs

### Split Assignment

Splits are assigned deterministically based on a composite record key:

```python
record_key = f"{fen}:{ply}"
split_bucket = int(sha256(record_key)[:8], 16) % 100
```

**Split allocation:**
- `0-4` → `frozenEval` (5%)
- `5-14` → `val` (10%)
- `15-99` → `train` (85%)

### Shard Naming and Size Limits

Shards use sequential deterministic naming:
- `shard_000.jsonl`
- `shard_001.jsonl`
- `shard_002.jsonl`
- ...

**Shard size limits:**
- Each shard contains at most `--shard-size` records (default: 10000)
- Shards are filled sequentially: when `shard_000` reaches the limit, records roll to `shard_001`
- The last shard may contain fewer records than the limit
- Shard count is determined dynamically based on total records and shard size

### Hash Computation

- **Shard hash**: SHA-256 of the complete shard file content
- **Assembly config hash** (v2): SHA-256 of canonical JSON representation of assembly parameters (shard size, filters, etc.)
- **Dataset digest** (v2): SHA-256 of canonical JSON combining:
  - Assembly config hash
  - Sorted list of input digests (from receipts or computed from PGN files)
  - Schema version identifiers
- **Filter config hash** (v1/v2): Legacy hash for backward compatibility

**Input digest computation:**
- For ingest receipts: Uses `receipt.artifact.sha256` (or `receipt.derived.decompressed_sha256` if decompressed)
- For PGN files: SHA-256 of file content with normalized line endings (`\r\n` → `\n`, `\r` → `\n`)

## Reproducibility

To reproduce a dataset exactly:

1. Use the same input sources (same PGN files or same ingest receipts with same content)
2. Use the same build configuration (same CLI arguments or config dict, including `shard_size`)
3. Use a frozen timestamp (or ensure `generated_at` is identical)

The same inputs + same config + same timestamp → **byte-identical outputs**.

**Receipt-based builds provide stronger reproducibility guarantees:**
- Receipts include SHA-256 hashes of artifacts
- Receipts document provenance (source URI, timestamps, tool versions)
- Dataset manifests include input digests, enabling verification of input integrity

## Schema Validation

Every record in a shard validates against the **Context Bridge schema (v1)**. The manifest validates against the **Dataset Manifest schema (v2)** (or v1 for legacy datasets).

Validation is enforced in tests and can be performed manually:

```python
import json
import jsonschema

# Validate a shard record
schema = json.load(open("src/renacechess/contracts/schemas/v1/context_bridge.schema.json"))
with open("datasets/sample/shards/shard_000.jsonl") as f:
    for line in f:
        if line.strip():
            record = json.loads(line)
            jsonschema.validate(record, schema)

# Validate manifest
manifest_schema = json.load(open("src/renacechess/contracts/schemas/v1/dataset_manifest.v2.schema.json"))
manifest = json.load(open("datasets/sample/manifest.json"))
jsonschema.validate(manifest, manifest_schema)
```

## Receipt-Based Building (M03+)

**Why use receipts?**

Receipt-based building provides:
- **Auditable provenance**: Full traceability from source to dataset
- **Integrity verification**: SHA-256 hashes ensure input authenticity
- **Reproducibility**: Receipts document exact inputs and tool versions
- **Decompression handling**: Automatic use of decompressed artifacts when available

**Receipt requirements:**
- Receipts must point to decompressed PGN files (or have `derived.decompressed_path`)
- If a receipt points to a `.zst` file without a decompressed path, dataset build will fail with a clear error message
- Re-run `renacechess ingest ... --decompress` to create receipts with decompressed paths

**Path resolution:**
- Absolute paths in receipts are used directly
- Relative paths are resolved relative to `--cache-dir` (if provided) or the receipt file's directory
