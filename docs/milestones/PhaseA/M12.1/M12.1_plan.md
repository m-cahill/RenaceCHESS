# M12.1 — Pydantic Compatibility Restoration (PYDANTIC-COMPAT-001)

**Type:** Correctness Restoration  
**Phase:** A — Post-PoC Hardening  
**Position:** Blocking M12 closeout  
**Parent Milestone:** M12 — Audit Remediation Pack

---

## Objective

Restore compatibility between RenaceCHESS's Pydantic models and pinned Pydantic v2 behavior **without changing PoC semantics**.

### Why This Milestone Exists

M12's dependency pinning (`pydantic~=2.0.1` → resolves to 2.0.3) correctly revealed a latent compatibility issue:

- **Issue:** Pydantic models use `validate_by_alias=True`, which requires camelCase aliases in dict inputs
- **Reality:** Codebase uses snake_case field names for model instantiation
- **Impact:** 175 test failures, 196 MyPy errors
- **Root Cause:** Semantic drift masked by floating dependency versions (`pydantic>=2.0.0`)

This is **not a bug introduced by M12** — it is a **pre-existing correctness issue correctly revealed by supply-chain hardening**.

### Why This Is Valid

- This is **not "new functionality"** — it is restoring correctness under deterministic dependencies
- The **semantic meaning** of the models does not change
- Only **instantiation mechanics** change (alias vs field name)
- PoC contracts remain **unchanged**

## Deliverables

### 1. Compatibility Strategy Decision

**Choose and document ONE approach:**

**Option A: Allow Both (Recommended)**
- Configure models to accept both snake_case field names AND camelCase aliases
- Use `populate_by_name=True` in model config (if not already present)
- Verify `validate_by_alias=True` behavior is preserved for JSON serialization

**Option B: Standardize to Aliases**
- Update all model instantiation to use camelCase aliases
- Keep `validate_by_alias=True` for JSON contract compliance
- More invasive but cleaner long-term

**Decision Criteria:**
- Minimal code changes
- Preserves existing call patterns where possible
- Maintains JSON contract compliance (camelCase output)
- Clear, auditable rule

### 2. Model Configuration Updates

- Update Pydantic model configs to implement chosen strategy
- Ensure JSON serialization still uses camelCase aliases (contract requirement)
- Document the chosen approach in model docstrings

### 3. Call Site Updates

- Update all model instantiation sites to match chosen strategy
- Fix all 175 test failures
- Fix all 196 MyPy errors
- Ensure no semantic changes to model behavior

### 4. Regression Tests

- Add tests verifying alias behavior works correctly
- Add tests verifying field name behavior works correctly (if Option A)
- Ensure tests fail if alias/field name behavior breaks

### 5. Documentation

- Document the chosen compatibility strategy
- Update relevant governance docs if needed
- Add note to M12.1 summary explaining the fix

### Investigation Steps

1. **Audit all Pydantic model definitions:**
   - Check current `model_config` settings
   - Identify which models have `validate_by_alias=True`
   - Identify which models have `populate_by_name=True`

2. **Audit all instantiation patterns:**
   - Find all `ModelName(...)` constructor calls
   - Categorize by instantiation style (dict, kwargs, etc.)
   - Count how many use snake_case vs camelCase

3. **Choose strategy:**
   - Evaluate Option A vs Option B
   - Consider code change volume
   - Consider long-term maintainability

4. **Implement fix:**
   - Update model configs
   - Update call sites
   - Run tests incrementally

5. **Validate:**
   - Full CI run
   - Verify JSON output still uses camelCase
   - Verify no semantic changes
